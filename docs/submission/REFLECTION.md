# Coding Agent TUI 反思报告

> 231870260 蒋程浚

## 哪些 Superpowers 技能在你的项目里发挥了最大作用？哪些感觉是“形式大于实质”？

我认为发挥作用最大的能力如下：

**brainstorming** 在发现隐形需求方面帮助较大。相比传统开发中由开发者自行穷举需求，智能体能够较快补充未被显式表达但实际会影响系统行为的边界情况。在 TUI 系统中，很多问题并不会直接出现在功能需求中，而是在交互过程中逐渐暴露。brainstorming 的价值在于帮助开发者更早意识到这些潜在问题，从而降低后续返工成本。

**systematic-debugging** 对复杂状态系统尤其重要。智能体默认倾向于“猜测式修复”，即通过快速修改代码不断尝试解决问题，但这种方式在状态复杂、交互较多的系统中容易引入新的错误。相比之下，systematic-debugging 强调问题复现、假设验证与逐步定位，使 debug 从经验行为变成一种结构化流程。对于 AI 协作而言，这种约束能够有效减少随机试错。

**spec-driven** 的价值在 AI coding 环境下被进一步放大。传统开发中，模糊需求往往可以通过人与人之间的沟通逐渐澄清，但智能体在面对模糊规格时更倾向于主动补全需求。因此，spec 不只是需求描述，更是一种边界约束机制，其价值在于明确系统能力范围、限制过度推断。

相比之下，我认为 **writing-plan** 与 **TDD** 在当前 workflow 中存在一定程度的“形式大于实质”。

当前的 **writing-plan** 往往已经包含较多实现细节，甚至接近完整代码逻辑，这使得 plan 更像 implementation document，而不是设计工具。一方面，过长且过细的计划难以 review；另一方面，当计划已经精确到实现层时，subagent 的自主设计空间被压缩，其角色更接近 executor。

对于 **TDD**，我认为其在 AI 协作中的角色发生了一定变化。传统 TDD 的核心在于由测试驱动设计，通过 failing test 明确能力边界与接口形式。但在当前 workflow 中，系统设计往往已经在 plan 阶段基本完成，因此测试并未真正参与设计，而更像一种质量验证机制。从这一点来看，subagent 中的 TDD 流程与 code and test 的区别并不明显。

## TDD 强制在 AI 协作下的体感如何？它是阻碍还是放大器？

我认为 TDD 在 AI coding 中既不是单纯的阻碍，也不完全是放大器，而是发生了角色变化。

传统 TDD 的隐含假设是：开发者尚未确定具体实现，因此通过 failing test 驱动接口设计，并通过小步迭代逐渐完善系统。但在当前 Superpowers workflow 中，系统设计往往已经在 plan 阶段较完整地形成，甚至包含部分实现思路，因此测试并没有真正参与设计过程。

在这种情况下，TDD 更接近一种回归测试与质量保障机制，而不再是设计驱动工具。其价值主要体现在稳定接口或高回归风险模块中，但对于探索性较强、需求持续变化的模块，其收益会下降。因此，我认为当前 workflow 中的 TDD 更像传统瀑布式开发后的 verification，而非敏捷开发语境下的 design driver。

## subagent-driven 工作流让智能体能自主跑多久不偏题？什么样的 task 拆解颗粒度最优？

我认为 subagent 的稳定性与任务上下文复杂度高度相关。对于目标明确、上下文独立的任务，智能体能够较长时间保持稳定执行；但对于跨模块、需求模糊或上下文依赖较强的任务，则更容易逐渐偏离目标并自行扩展需求。

因此，task 的拆分粒度非常重要。粒度过粗时，智能体容易发生偏移；粒度过细时，则会导致协调成本上升，使 subagent 更像一个被动执行脚本，而失去自主价值。

实践中，我认为更合理的粒度接近“可独立验证的模块级任务”，即输入、目标与验收标准明确，但保留一定实现空间。这也意味着，一个 spec 未必需要一次性完成全部实现，而更适合拆分为多个渐进式的plan，通过短周期反馈不断修正方向。

## SPEC 与 PLAN 的质量如何影响了后续实现质量？是否有过“规约不清晰导致 subagent 跑偏”的具体案例？

我认为 SPEC 与 PLAN 的作用并不相同：SPEC 决定方向是否正确，而 PLAN 决定执行效率。

在 AI coding 环境下，高质量 SPEC 的价值被进一步放大。由于智能体倾向于主动补全未定义内容，因此模糊的 specification 很容易导致过度实现或错误抽象。相比传统开发，问题定义错误的代价反而更高。

相比之下，PLAN 更影响 subagent 的执行质量。当前 workflow 中的 plan 往往偏向一次性完整规划，甚至已经包含较多实现细节。这种方式虽然能够提高短期确定性，但会降低灵活性，并使 subagent 缺少自主设计空间。

我认为更合理的方式是采用该策略：先完成高层目标与接口约束，再随着需求变化逐步补充后续计划，而不是一次性完成全部设计。这种方式更适合探索式项目，也更符合 AI coding 的快速迭代特点。

## 你设计的最有效的 prompt / context 策略是什么？为什么有效？

我使用的最有效的策略是接入外部 MCP（如 context7），并明确要求智能体使用该工具。

智能体在 coding 中的一个主要问题是“想当然实现”，即依据已有模式生成看似合理但并不符合官方规范的代码。相比不断优化 prompt 描述，我发现控制智能体的信息来源更有效。通过提供官方文档与标准示例，可以减少错误 API 使用、避免版本不兼容问题，并降低幻觉风险。

因此，我认为 prompt engineering 的关键并不只是描述得更复杂，而是如何提供高质量 context。

## 如果重做，你会改变哪些做法？

如果重新进行项目，我会调整 plan 的拆分方式。

当前 workflow 更接近“一个 spec 对应一个完整大 plan”，但对于探索性项目而言，需求往往会在开发过程中持续变化，因此完整规划容易造成返工。

更合理的方式是将一个 spec 拆分为多个渐进式 plan，通过短周期反馈不断修正目标。这既能够降低一次性 planning 成本，也能保留 subagent 的自主空间，使其真正参与设计，而不是单纯执行既定方案。

另外对于 log 的记录，我当前的方案是通过 prompt 约束，并非硬性要求，这导致我有几个操作记录失败，要让我手动添加。我可能会考虑配置 hook 强制检查是否需要记录 log

## 你对 Superpowers 这套方法论的批判：它假设了什么？这些假设在你的项目里成立吗？

我认为 Superpowers 方法论本身是偏敏捷开发的，其核心思想仍然强调快速反馈、持续迭代与任务拆分。例如 spec-driven、subagent 协作以及 code-review 等实践，本质上都在尝试降低单次决策风险，并通过更小粒度的任务推进开发。

但在实际 AI coding workflow 中，我认为部分环节容易出现一种“瀑布化倾向”，即某些步骤被过度前置。例如 writing-plan 在实践中容易演化为一次性的大计划，甚至提前包含较多实现细节；TDD 也可能从“测试驱动设计”转变为“实现后的验证工具”。结果是，虽然流程形式上仍然是敏捷的，但实际执行体验会逐渐接近轻量级瀑布开发。

我认为问题并不在于敏捷或瀑布本身孰优孰劣。对于需求稳定、边界明确的任务，较完整的 upfront planning 仍然是有效的；而对于探索性较强、需求不断变化的项目，则需要保留更强的迭代能力。本项目中，需求与交互逻辑需要持续试错，因此更适合短周期反馈与渐进式 planning。

因此，我对 Superpowers 的批判并不是否定其方法论，而是认为在 AI 协作场景下，需要避免某些实践被“工具能力”推向过度前置设计。方法论本身仍然成立，但需要根据项目特征动态调整强度与粒度。

## 你对当前 AI4SE 工具与方法论的整体看法。

我认为当前 AI4SE 工具正在快速演进，而方法论仍处于磨合阶段。一个明显现象是：AI coding 已经改变了传统软件工程的一些成本结构，但很多开发流程仍在沿用过去的实践。

例如，在传统开发中，代码实现成本较高，因此前置计划与风险控制是合理的；而在 AI coding 中，实现与试错成本显著下降，使得快速迭代变得更加可行。但与此同时，智能体又容易产生过度推断、上下文漂移与错误抽象，因此也不能完全放弃规范化流程。

因此，我认为未来的 AI4SE 不会简单走向“完全敏捷”或“完全瀑布”，而更可能是一种动态平衡：既保留敏捷开发中的快速反馈与小步迭代，也保留瀑布式方法在边界定义、系统约束与复杂任务规划上的优势。

换句话说，AI coding 并不是替代软件工程，而是在重新调整软件工程中不同方法的权重。未来开发者的重要能力，可能不再只是写代码，而是如何定义问题、约束智能体，并持续修正方向。
